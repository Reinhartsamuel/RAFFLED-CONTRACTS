# Mainnet Preparation Plan: RaffleManager4

## Phase 1: Critical Security Fixes (Must Do Before Testing)

### 1.1 Fix `setTrustedSigner` Access Control
**File:** `src/FreeEntryVerifier.sol:39`
```solidity
// BEFORE (vulnerable):
function setTrustedSigner(address _newSigner) external { ... }

// AFTER:
function setTrustedSigner(address _newSigner) external onlyOwner { ... }
```
**Why:** Currently anyone can replace the trusted signer and mint unlimited free entries.

### 1.2 Add `_trustedSigner` as Constructor Parameter
**File:** `src/RaffleManager4.sol` constructor (lines 163-194)
```solidity
// BEFORE:
constructor(address _vrfCoordinator, bytes32 _keyHash, uint256 _subId, address _paymentToken, address _treasury)
    ...
    FreeEntryVerifier(msg.sender)

// AFTER:
constructor(address _vrfCoordinator, bytes32 _keyHash, uint256 _subId, address _paymentToken, address _treasury, address _trustedSigner)
    ...
    FreeEntryVerifier(_trustedSigner)
```
**Why:** Deployer key should not automatically become the trusted signer. The trusted signer should be explicitly set (ideally a backend multisig or EOA).

### 1.3 Signature Validity — Rely on Raffle Expiry
**Decision:** Skip adding a `deadline` field to the EIP-712 payload (Option A).

**Rationale:** `enterFreeRaffle` already enforces `block.timestamp >= raffle.expiry` and reverts with `RaffleNotOpen`. This means signatures are **naturally bounded** by the raffle's on-chain expiry — they become useless once the raffle closes.

**Tradeoff:** If the trusted signer is rotated, old signatures for still-open raffles will remain redeemable. This is acceptable for v1; can add `deadline` later if tighter control is needed.

**No backend code changes required.**

### 1.4 Remove `manualFulfillWinnerByRandomWord` Entirely
**File:** `src/RaffleManager4.sol` lines 535-558

Delete the function completely. It is fully gameable since the caller supplies `_randomWord`. `manualFulfillWinner` (by index) already serves as the frontend fallback.

### 1.5 Enforce Minimum Raffle Duration
**File:** `src/RaffleManager4.sol` in `createRaffleERC20` and `createRaffleERC721`

Add constant and check:
```solidity
uint256 public constant MIN_DURATION = 2 hours;

// In both create functions:
if (_duration < MIN_DURATION) revert DurationTooShort(_duration, MIN_DURATION);
```
Add error: `error DurationTooShort(uint256 requested, uint256 minimum);`

### 1.6 Add `nonReentrant` to `performUpkeep`
**File:** `src/RaffleManager4.sol` line 437
```solidity
function performUpkeep(bytes calldata performData) external override nonReentrant {
```

### 1.7 Update Natspec Comments
**File:** `src/FreeEntryVerifier.sol` line 8

Change `/// @notice Tester contract to validate EIP-712 signature verification` to production-grade documentation like:
```solidity
/// @notice EIP-712 signature verifier for free raffle entries.
///         Validates backend-signed free entry claims and prevents double-claiming.
```

## Phase 3: Deployment Script

### 3.1 File: `script/Deploy4.s.sol`

**Pattern:** Follow `script/Deploy3.s.sol` structure but updated for RaffleManager4.

**Environment Variables Required (in `.env`):**
```bash
DEPLOYER_PRIVATE_KEY=
RPC_URL=
ETHERSCAN_API_KEY=

# VRF Configuration
VRF_COORDINATOR=
KEY_HASH=
SUB_ID=

# Business Logic
PAYMENT_TOKEN=
TREASURY=
TRUSTED_SIGNER=

# Optional: Initial platform fee
INITIAL_FEE_BPS=0
```

**Script Logic:**
1. Load all env vars
2. Validate critical addresses are non-zero
3. `vm.startBroadcast(deployerKey)`
4. Deploy `RaffleManager4` with all params
5. If `INITIAL_FEE_BPS > 0`, call `proposeFeeChange` and `applyFeeChange` (owner-only)
6. `vm.stopBroadcast()`
7. Log all deployed addresses and next steps

**Output:**
```
===========================================
RaffleManager4 deployed: 0x...
Payment Token:           0x...
Treasury:                0x...
Trusted Signer:          0x...
VRF Coordinator:         0x...
===========================================
Next steps:
1. Add RaffleManager4 as VRF consumer on vrf.chain.link
2. Fund VRF subscription with LINK
3. Set initial platform fee (if not already set)
4. Verify contract on Etherscan
===========================================
```

## Known Limitations / Future Improvements

1. **`checkUpkeep` linear scan** – Will break at scale (hundreds of raffles). Future: implement active raffle queue or permissionless close mechanism.
2. **`freeEntryClaimed` mapping grows unbounded** – Storage grows with each unique `(raffleId, user)` pair. Consider periodic cleanup or separate verifier contract.
3. **Malicious ERC-20 tokens** – No allowlist for prize assets. Host could use tokens with transfer hooks/blacklists.
4. **Fee timelock gaming** – Owner can repeatedly reset timelock. Consider blocking overlapping proposals.
5. **No pause mechanism** – Consider adding `Pausable` from OpenZeppelin for emergency stops.
